**后端负责高频的矿石数据管理与详细日志存储，链端负责核心资产（卡片/勋章）的权属证明。**

在这种架构下，勋章不再是一个孤立的图片，而是一个指向后端详细成长档案的**“链上索引”**。

---

## 一、 整体技术方案：混合驱动架构

* **矿石与日志 (Off-chain)**：存储在 **MongoDB** 中，记录每一笔矿石产生的具体对话内容、时间、类别和数值。
* **资产与存证 (On-chain)**：存储在 **Avalanche C-Chain** 上。
* **追溯桥梁 (The Bridge)**：通过勋章 NFT 的 `tokenURI` 或 `Metadata` 中的 `UID` 链接到后端的详细 API 接口。

---

## 二、 详细技术路线

### 1. 后端与数据库 (Node.js + MongoDB)
* **数据模型设计**：
    * **User Schema**: 存储钱包地址、各类别矿石总额。
    * **OreLog Schema**: 存储单次记录。字段包括：`userId`, `content` (用户写的日记), `category`, `oreAmount`, `timestamp`, `medalId` (如果该矿石已被合成，则关联勋章 ID)。
* **API 接口**：
    * `POST /api/record`: 接收记录，增加矿石，写入日志。
    * `GET /api/medal/:id/trace`: **关键接口**。根据勋章 ID，回溯查询所有支撑该勋章生成的原始矿石记录。

### 2. 链端合约 (Solidity on Avalanche Fuji)
* **ERC-1155 (卡片)**：作为中间态资产。
* **ERC-5192 (勋章/SBT)**：
    * **Mint 逻辑**：前端调用 `mintMedal(string memory metadataURI)`。
    * **Metadata 设计**：`metadataURI` 指向一个存储在 IPFS 或服务器上的 JSON 文件，其中包含一个指向后端追溯接口的 `external_url`。

### 3. 前端交互 (Next.js + Tailwind)
* **记录环节**：用户提交后，前端先请求后端 API 更新矿石，完成后在 UI 弹窗提示“矿石已存入矿洞”。
* **追溯环节**：用户点击勋章墙上的某个勋章，前端根据勋章的 `TokenID` 调用后端的 `trace` 接口，以时间轴形式展示出背后的每一条日记。
